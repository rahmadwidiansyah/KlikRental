# Audit Migration
Berikut adalah hasil audit teknis domain bisnis berdasarkan struktur *Migration* dan *Model* yang Anda berikan.

### Domain Model

**Entitas Utama & Relasi Antar Tabel:**

* **User (Pengguna):** Pusat identitas. Memiliki relasi ke `Booking` (sebagai penyewa) dan `Review` (sebagai pemberi ulasan).
* **Vehicle (Armada):** Memiliki relasi *One-to-Many* ke `VehicleImage` (Galeri foto). Memiliki status ketersediaan dan klasifikasi (Standard/Premium/VIP).
* **Driver (Supir):** Entitas mandiri (opsional dalam penyewaan). Memiliki tarif harian dan status ketersediaan.
* **Zone (Zona Lokasi):** Digunakan ganda dalam relasi *Booking* (sebagai `pickup_zone_id` dan `dropoff_zone_id`). Mendukung titik koordinat (Lat/Long) dan penanda lokasi kantor (`is_office`).
* **Promo (Diskon):** Entitas mandiri, digunakan opsional dalam *Booking*.
* **Booking (Reservasi):** Pivot transaksi utama (Transaksional). Menghubungkan `User`, `Vehicle`, `Driver` (opsional), `Zone` (Jemput/Antar), dan `Promo` (opsional).
* **Payment (Pembayaran):** Relasi *One-to-One* (Asumsi bisnis) atau *One-to-Many* dengan `Booking`. Menyimpan jejak transaksi *Payment Gateway*.
* **Review (Ulasan):** Relasi spesifik ke `Booking` dan `User`.
* **TeamMember:** Entitas statis, tampaknya digunakan untuk halaman *Company Profile* atau *Landing Page* (tidak terelasi dengan alur penyewaan utama).

### Business Workflow

**Fitur yang Benar-Benar Diimplementasikan:**

* **Smart Avatar System:** Logika untuk membedakan *display picture* berdasarkan sumber (URL Google, Storage lokal, atau inisial UI-Avatars otomatis). Terintegrasi dengan antarmuka bawaan Filament.
* **Perhitungan Pajak Terpusat:** *Booking* secara *default* membebankan tarif pajak 11% (`tax_rate`) yang dipisahkan kalkulasinya pada kolom `tax_amount`.
* **Otomasi Status Inventaris (Event Driven):** Sinkronisasi otomatis ketersediaan Mobil dan Supir berdasarkan perubahan status `Booking` menggunakan fitur Eloquent `booted()`.
* **Sistem Rating Multi-Aspek:** Penilaian dipecah menjadi 3 aspek spesifik: Kondisi Mobil, Pelayanan Perusahaan, dan Kinerja Supir (opsional).
* **Manajemen Zona Kompleks:** Pemisahan antara zona lokasi *custom* dengan alamat kantor cabang (via *flag* `is_office`) beserta integrasi tautan Google Maps.

### User Roles

* **`admin`:** Memiliki hak istimewa berdasarkan *method* `canAccessPanel(Panel $panel)`. Dapat masuk ke *dashboard* Filament.
* **`customer`:** *Role default* untuk pendaftar publik/Google OAuth. Tidak memiliki akses panel.
* **`staff`:** Tersedia di *Migration* (sebagai nilai Enum), namun tidak ada logika pembatasan hak akses yang spesifik di model yang diberikan.

### Booking Lifecycle

**Status Transaksi & Alur Hidup (`Booking`):**

1. **`pending`:** Status awal saat data penyewaan dibuat.
2. **`paid`:** Pembayaran telah diselesaikan (menunggu waktu pengambilan).
3. **`in_use`:** Kendaraan diserahkan ke pelanggan. **(Otomasi memicu:** Status `Vehicle` menjadi `rented`, `Driver` menjadi `on_duty`).
4. **`late`:** Melewati tenggat waktu pengembalian. (Memicu perhitungan `late_fee`).
5. **`completed`:** Penyewaan selesai normal. **(Otomasi memicu:** Status `Vehicle` dan `Driver` kembali `available`).
6. **`cancelled`:** Transaksi batal. **(Otomasi memicu:** Status inventaris kembali `available`).

### Payment Flow

* Alur tabel sudah disiapkan untuk integrasi Midtrans.
* Menyimpan ID transaksi dari pihak ketiga (`transaction_id`), tipe pembayaran, total kotor (`gross_amount`), dan waktu penyelesaian (`settlement_time`).
* Status awal `transaction_status` bernilai `pending`.

### Review System

* Pelanggan hanya bisa memberikan 1 set ulasan per ID Transaksi (terhubung langsung ke `booking_id`).
* Nilai `driver_rating` dapat dilewati (*nullable*) yang mengindikasikan sewa mobil lepas kunci.

# Audit Filament
Berikut adalah hasil audit teknis untuk panel administrasi (Filament V4) berdasarkan struktur *Resource*, *Page*, *Schema*, *Table*, dan *Widget* yang Anda lampirkan:

### Dashboard Features

Berdasarkan modul *Widgets* yang diimplementasikan:

* **Stats Overview (Shortcut Cerdas):** 4 Kartu metrik utama (Order Hari Ini, Order Dibayar, Mobil Disewa, Omset Bulan Ini). Tiga kartu pertama memiliki URL tautan yang secara otomatis menerapkan *filter* pada tabel `BookingResource` saat diklik.
* **Tabel Transaksi Terbaru (Latest Bookings):** Menampilkan 5 transaksi terakhir langsung di halaman *dashboard*. Admin dapat langsung mengubah **Status Transaksi** (*inline editable select*) tanpa harus masuk ke halaman detail/edit.
* **Grafik Pendapatan (Revenue Chart):** Grafik garis (*Line Chart*) interaktif yang memetakan total omset kotor bulanan selama tahun berjalan (hanya menghitung transaksi berstatus `completed`).
* **Grafik Ketersediaan (Vehicle Status):** Grafik donat (*Doughnut Chart*) yang memvisualisasikan rasio status seluruh armada (Tersedia, Disewa, Perawatan) secara *real-time*.

### Data Management

Berdasarkan *Resource* data master:

* **Manajemen Armada (VehicleResource):** CRUD data mobil dengan spesifikasi detail (tipe, transmisi, kursi, bagasi). Mendukung unggah *Multiple Images* (Galeri) menggunakan `Repeater` dengan *toggle* `is_primary` untuk menentukan foto *thumbnail* utama.
* **Manajemen Supir (DriverResource):** Pengelolaan data supir, nomor HP, foto profil, dan status penugasan. Menggunakan *fallback* API UI-Avatars otomatis jika foto tidak diunggah.
* **Manajemen Zona & Lokasi (ZoneResource):** Mengelola titik lokasi beserta biaya tambahan (`additional_cost`). Terdapat pemisahan logika: Zona biasa (titik jemput/antar) vs Kantor Cabang (`is_office`) yang memiliki *field* alamat, *Latitude*, dan *Longitude*.
* **Manajemen Promo (PromoResource):** Pembuatan kode diskon berdasarkan persentase dengan fitur *capping* atau batas maksimal nominal diskon (`max_discount`) dan masa berlaku.
* **Manajemen Tim Pengembang (TeamMemberResource):** Data statis untuk profil nama, peran (*role*), dan foto (kemungkinan untuk *Company Profile* di halaman pelanggan).

### Operational Features

* **Pencarian Pintar (Smart Search):** Pada tabel Transaksi (Booking), admin dapat mencari data pemesanan secara fleksibel menggunakan **Nama Mobil** atau **Plat Nomor** (pencarian menembus relasi tabel).
* **Auto-Generate Plat Nomor Lokal:** Saat menambah armada baru, sistem secara otomatis mengisi *placeholder* dan membuatkan plat nomor acak wilayah Semarang (H + 4 Angka + 2/3 Huruf) jika admin tidak mengisinya, dan memaksa format ke Huruf Kapital (*Capitalize*).
* **Status Toggling Cepat:** Pada tabel Zona, admin dapat langsung mengaktifkan/menonaktifkan zona (*is_active*) atau mengubah status kantor (*is_office*) melalui *Toggle Column* tanpa perlu masuk ke form *Edit*.

### Business Administration Features

* **Pajak Bawaan (Default Tax):** Form `Booking` secara bawaan membebankan tarif pajak 11% (`tax_rate`) pada setiap pembuatan transaksi manual oleh Admin.
* **Manajemen Siklus Pesanan Terpusat:** Admin memegang kendali penuh pada `BookingResource` untuk mengubah status alur penyewaan (*Pending, Paid, In Use, Late, Completed, Cancelled*).
* **Kalkulasi Omset Aman:** Dasbor *Analytics* didesain kebal terhadap pesanan batal atau pesanan fiktif, karena `RevenueChart` dan metrik 'Omset Bulan Ini' hanya menjumlahkan `total_price` dari pesanan yang statusnya sudah `completed`.

# Audit Controllers
Berikut adalah hasil audit teknis berdasarkan kode Controller, Console Command, dan Form Request yang Anda lampirkan:

### Controller Map

| Nama Controller | Modul Utama | Fungsi Utama yang Diimplementasikan |
| --- | --- | --- |
| **`HomeController`** | Katalog & Dasbor | Menampilkan katalog kendaraan dengan filter (*class*, *type*, tanggal), logika pengecekan mobil yang sedang disewa "hari ini", dan data statistik. |
| **`BookingController`** | Transaksi Reservasi | Validasi *double-booking*, *endpoint* AJAX kalkulasi harga *real-time*, pembuatan *invoice*, integrasi Midtrans Snap Token, dan *submit* ulasan. |
| **`MidtransController`** | Pembayaran (*Webhook*) | Menangkap *callback* Midtrans, memperbarui status pesanan (`paid`, `pending`, `cancelled`), dan memicu *Webhook* n8n khusus pesanan sukses. |
| **`DriverController`** | Profil Supir | Menampilkan daftar supir beserta jumlah transaksi dan rata-rata *rating*. |
| **`ProfileController`** | Manajemen Akun | Pembaruan data identitas operasional (KTP, SIM, NIK, No HP, Alamat) dan Avatar profil. |
| **`GoogleController`** | Autentikasi SSO | *Login/Register* otomatis via Akun Google dan sinkronisasi Avatar Google. |
| **`BookingMonitor`** | Otomasi Latar Belakang | *Cron job* untuk pengingat jadwal ambil (H-30m), eskalasi admin (H+10m), pengingat kembali (H-2 jam), dan pinalti *overtime*. |

---

### Business Flow

#### 1. Customer Journey

1. **Eksplorasi:** Pelanggan mengakses beranda/dasbor (`HomeController@index`) untuk melihat katalog. Fitur filter tanggal secara otomatis akan menyembunyikan mobil yang status jadwalnya beririsan (*overlap*).
2. **Kalkulasi:** Di halaman form *booking*, pelanggan memilih jadwal, zona jemput/antar, supir (opsional), dan promo. Sistem menghitung secara otomatis via AJAX (`BookingController@calculatePrice`).
3. **Pemesanan:** Data divalidasi dengan ketat di *backend*. Jika lolos, status awal menjadi `pending`.
4. **Pembayaran:** Dialihkan ke halaman *Invoice* yang memuat Midtrans Snap Token untuk pembayaran mandiri.

#### 2. Booking Workflow

* **Proteksi Ganda (*Double-Booking*):** Sistem melakukan pengecekan ketersediaan jadwal tidak hanya pada `Vehicle` (Mobil), tetapi juga pada `Driver` (Supir) untuk memastikan tidak ada bentrok jadwal di tanggal yang sama.
* **Otomasi Pengingat (via `BookingMonitor`):**
* **H-30 Menit:** Mengirim *webhook* notifikasi persiapan ke pelanggan dan supir.
* **H+10 Menit (Eskalasi):** Jika jadwal sudah lewat 10 menit tapi status belum diubah admin menjadi `in_use`, sistem mengirim peringatan ke Admin.
* **H-2 Jam:** Mengirim pengingat waktu pengembalian.
* **Overtime (Lewat 30 Menit):** Otomatis mengubah status *booking* menjadi `late`, menghitung estimasi denda otomatis (Rp 50.000 / Jam), dan menembak *webhook* pinalti.



#### 3. Payment Workflow

* Sistem menggunakan Midtrans Snap Token yang di-*generate* saat status pesanan masih `pending`.
* Saat pelanggan membayar, `MidtransController@handleNotification` menerima *payload*.
* Jika transaksi sukses (`capture/settlement`), status *booking* diubah menjadi `paid`.
* **Proteksi Spam Webhook:** *Webhook* ke n8n (`booking_paid`) HANYA ditembakkan jika ada perubahan status dari *selain* `paid` menjadi `paid`.

#### 4. Authentication Flow

* **Role Routing:** Setelah *login* reguler, sistem mengecek *role*. Admin dilempar ke `/admin`, sedangkan Pelanggan ke `/dashboard`.
* **Pendaftaran Reguler:** Modifikasi pada `RegisteredUserController` mewajibkan pengisian `phone_number` saat *register*.
* **SSO Google:** Jika pelanggan masuk via Google dan *email* belum terdaftar, sistem akan membuatkan akun secara otomatis dengan *password* acak (*bypass* form *register*).

#### 5. Review Workflow

* Terletak di `BookingController@storeReview`.
* **Prasyarat:** Hanya bisa diisi jika status pesanan sudah `completed`.
* **Proteksi:** Terdapat pengecekan `exists()` agar pelanggan tidak bisa memberi ulasan lebih dari satu kali untuk satu kode *booking*.
* **Penilaian Multi-Aspek:** Pelanggan memberikan 3 *rating* terpisah (Kendaraan, Perusahaan, dan Supir). *Rating* supir bersifat opsional (`nullable`).

#### 6. Company Profile Features

* Halaman Publik menampilkan daftar Supir aktif beserta rata-rata ulasannya (`DriverController`).
* Data *Customer Service* (`/cs`) mengambil zona-zona yang secara spesifik ditandai sebagai `is_office = true`.

# Audit Views
Berikut adalah hasil analisis *frontend* (antarmuka pelanggan) berdasarkan kode Blade yang Anda berikan:

### Vehicle Catalog Features

Katalog kendaraan dirancang untuk memberikan informasi sekilas yang padat dan interaktif bagi pelanggan.

* **Visual Status Indikator:** Setiap kartu mobil dilengkapi *badge* status *real-time* (Tersedia hijau, Disewa biru, Perawatan merah).
* **Informasi Inti:** Menampilkan gambar utama (*thumbnail*), Nama Mobil, Tipe (SUV/MPV/dll), Transmisi, dan Harga Sewa per hari.
* **Dynamic CTA (Call To Action):** Tombol aksi menyesuaikan dengan ketersediaan mobil. Jika tersedia, tombol mengarahkan ke "Pesan Sekarang" dengan sorotan warna utama (*primary*). Jika tidak, tombol berubah menjadi "Lihat Detail" dengan warna netral.
* **Navigasi Data:** Dilengkapi dengan fitur *Pagination* bawaan Laravel untuk mengelola daftar kendaraan yang panjang, serta penanganan *Empty State* ("Belum ada kendaraan di kategori ini").

### Vehicle Detail Features

Halaman detail memberikan spesifikasi komprehensif dan membangun kepercayaan pelanggan sebelum menyewa.

* **Galeri Interaktif:** Menampilkan gambar utama yang dapat berubah secara dinamis saat pelanggan mengklik *thumbnail* gambar tambahan di bawahnya (menggunakan *Vanilla JavaScript*).
* **Spesifikasi Detail:** Menjabarkan kapasitas spesifik kendaraan seperti jenis Transmisi, Kapasitas Kursi, Kapasitas Bagasi (Koper), dan Jenis Bahan Bakar.
* **Status Ketersediaan Real-Time:** Menampilkan *badge* status dengan efek animasi *pulse* (berkedip) untuk mobil yang tersedia atau disewa.
* **Sistem Ulasan (Review):** * Menampilkan rata-rata *rating* (bintang) dan total ulasan.
* Menampilkan daftar ulasan individual lengkap dengan avatar pelanggan (*fallback* ke UI-Avatars jika tidak ada foto), waktu ulasan (`diffForHumans`), dan komentar.
* Memiliki fitur *Toggle* "Lihat Semua Ulasan" untuk menyembunyikan/menampilkan ulasan jika jumlahnya lebih dari 3.


* **Primary CTA:** Tombol "Sewa Sekarang" yang besar dan mencolok. Jika mobil sedang disewa/perawatan, tombol ini secara otomatis di-*disable* (tidak bisa diklik).

### Customer Experience

Pengalaman pelanggan didukung oleh transparansi informasi dan kemudahan akses bantuan operasional.

* **Peta Lokasi Interaktif:** Halaman CS dilengkapi peta interaktif menggunakan **Leaflet.js** yang dibatasi (*locked bounds*) secara spesifik hanya untuk area Semarang. Peta ini menampilkan *marker* dinamis untuk setiap kantor cabang yang terdaftar.
* **Pusat Bantuan Multi-Channel:** Akses instan (sekali klik) menuju WhatsApp CS, Telepon Darurat, dan Email Support.
* **FAQ Interaktif:** Menggunakan **Alpine.js** (`x-data`, `x-show`, `x-collapse`) untuk membuat daftar pertanyaan umum (Syarat lepas kunci, denda, metode pembayaran) yang bisa di-Buka/Tutup (*accordion*).
* **Transparansi Aturan (Legal):** Halaman Syarat & Ketentuan dan Kebijakan Privasi menjabarkan aturan secara eksplisit, seperti kewajiban KTP/SIM A, *Grace Period* 30 menit, denda keterlambatan Rp 50.000/jam, dan kebijakan *No Refund*.
* **Company Profile:** Halaman *About* menampilkan profil Tim Pengembang (Kelompok 6) beserta dosen pengampu untuk membangun kredibilitas proyek.

# Audit N8N 
Berikut adalah hasil analisis *workflow* otomatisasi **n8n** untuk sistem notifikasi WhatsApp (KlikRental) berdasarkan konfigurasi JSON yang Anda berikan:

### Arsitektur Otomatisasi Notifikasi (Event-Driven)

*Workflow* ini dirancang menggunakan pola *Event-Driven Architecture*. Semua notifikasi dipicu oleh satu *endpoint* Webhook (`POST /booking-paid`), dan n8n bertugas sebagai *router* utama yang mendistribusikan pesan berdasarkan tipe *event* (`{{ $json.body.event }}`).

Sistem ini terintegrasi langsung dengan WhatsApp menggunakan *node* **WAHA (WhatsApp HTTP API)**.

#### Diagram Alir Otomatisasi (ASCII)

```text
[Webhook: POST /booking-paid]
      │
      ▼
[Switch: Pengecekan Tipe Event]
 ├──> 1️⃣ booking_paid ──────────────> (Wait) ─┬─> WA: Admin (Info Pesanan Baru)
 │                                            ├─> WA: Customer (Bukti Bayar Berhasil)
 │                                            └─> [Cek Driver] ──(Ya)──> WA: Driver (Jadwal Tugas)
 │
 ├──> 2️⃣ booking_reminder_2_hours ──> [Cek Driver] ─┬─> (Ya) ──> WA: Driver (Pengingat Sisa 2 Jam)
 │                                                  └─> (Tidak) ──> WA: Customer (Pengingat 2 Jam)
 │
 ├──> 3️⃣ booking_late ──────────────> (Wait) ─┬─> WA: Admin (Alert Overtime)
 │                                            └─> [Cek Driver] ─┬─> (Ya) ──> WA: Driver (Info Overtime)
 │                                                              └─> (Tidak) ──> WA: Customer (Info Denda)
 │
 ├──> 4️⃣ booking_pickup_reminder ───> (Wait) ──> [Cek Driver] ─┬─> (Ya) ──> WA: Driver (Siap 30 Menit)
 │                                                              └─> (Tidak) ──> WA: Customer (Jadwal 30 Menit)
 │
 └──> 5️⃣ booking_pickup_escalation ─> WA: Admin (Eskalasi: 10 Menit Telat Serah Terima)

```

---

### Analisis Logika & Flow

Terdapat 5 *flow* utama yang ditangani oleh *workflow* ini dengan tingkat *routing* yang sangat spesifik dan efisien:

* **Flow 1 (Booking Paid):** Saat pembayaran lunas, sistem secara paralel mengirim pesan ke Pelanggan dan Admin. Terdapat logika bersyarat (`Switch` node) yang mengecek variabel `has_driver`. Jika pelanggan menyewa beserta supir, *node* akan mencabangkan satu pesan tambahan khusus untuk Driver.
* **Flow 2 & 4 (Pengingat Jadwal):** Flow ini menangani *reminder* (30 menit sebelum mulai dan 2 jam sebelum selesai). Logikanya sangat cerdas: jika *order* menggunakan supir (`has_driver: true`), pesan peringatan dikirim ke **Supir** agar bersiap. Jika lepas kunci (`has_driver: false`), pesan langsung ditargetkan ke **Pelanggan**.
* **Flow 3 (Keterlambatan/Overtime):** Ketika durasi sewa habis, Admin adalah pihak pertama yang langsung mendapat notifikasi berserta estimasi denda (`current_late_fee`). Paralel dengan itu, sistem menegur Pelanggan (jika lepas kunci) atau Supir (agar mengonfirmasi status).
* **Flow 5 (Eskalasi Serah Terima):** Jika pelanggan/supir telat melakukan serah terima lebih dari 10 menit dari waktu *start*, notifikasi merah (Eskalasi) langsung ditembakkan ke Admin untuk ditindaklanjuti secara manual agar sistem tidak terblokir.

---

### Teknik & Format Data (Best Practices)

1. **Format Nomor Telepon Dinamis (Regex):**
Di dalam *node* WAHA, Anda menggunakan *expression* yang sangat solid untuk menangani berbagai macam input nomor telepon Indonesia (misal: `0812...`, `62812...`, atau `+62812...`):
`{{ $json.body.customer_phone.toString().replace(/\D/g, '').replace(/^0/, '62') + '@c.us' }}`
*Script* ini secara otomatis membuang karakter non-angka (seperti spasi atau tanda plus), lalu mengganti angka `0` di depan menjadi `62`, dan mengakhirinya dengan format Wha WhatsApp `@c.us`. Ini mencegah pesan gagal terkirim (Bounced).
2. **Templating Pesan Kondisional:**
Pada notifikasi Admin dan Customer, Anda menggunakan *inline ternary operator* untuk merender teks secara dinamis tanpa perlu membuat *node* terpisah.
Contoh: `{{ $json.body.has_driver ? 'Pakai Supir (' + $json.body.driver_name + ')' : 'Lepas Kunci (Tanpa Supir)' }}`. Ini membuat *workflow* tetap ringkas dan tidak memakan memori n8n.
3. **Penggunaan "Wait" Node (Rate Limiting Control):**
Terdapat beberapa *node* `Wait` sebelum proses *broadcast* ke banyak pihak sekaligus (seperti pada Flow 1). Meskipun durasinya diset minimal (`amount: 1`), ini adalah praktik yang bagus untuk mencegah antrean API WhatsApp (WAHA) membludak di waktu yang sama, sehingga mengurangi risiko akun terdeteksi sebagai *spam/bot* berlebihan.

### 1. Development Environment

Lingkungan lokal dirancang khusus untuk kenyamanan *coding* dengan pembaruan instan (*hot-reloading*).

* **Live Sync:** Menggunakan *bind mount* (`.:/var/www`) pada `docker-compose.yml` (dev), sehingga setiap perubahan kode PHP langsung terdeteksi tanpa perlu *build* ulang *container*.
* **Vite HMR:** Mengekspos port `5173` memungkinkan *Hot Module Replacement* untuk aset *frontend* berjalan mulus di *browser* lokal.
* **Database Aksesibel:** Port MySQL `3306` diekspos ke *host* agar database mudah diakses dan dikelola menggunakan *tools* GUI seperti DBeaver atau HeidiSQL.
* **Image Build:** Melakukan *build* dari Dockerfile lokal yang sama, memastikan konsistensi *environment* antara dev dan prod (Apache + PHP 8.4).

### 2. Production Environment

Fokus utama di sini adalah stabilitas, keamanan, dan *persistency*.

* **Immutable Image:** Tidak ada *build* kode secara lokal. *Service* aplikasi menarik *image* yang sudah dikompilasi secara final dari GitHub Container Registry (GHCR).
* **Storage Persistency:** Memisahkan *state* dari *container*. Volume `./storage` di-*mount* ke `/var/www/storage` untuk memastikan data unggahan (seperti KTP/SIM) dan log tidak hilang saat *container* diperbarui. Berkas `.env` juga disuntikkan secara dinamis dari luar.
* **CasaOS Integration:** Kehadiran metadata `x-casaos` menunjukkan arsitektur ini sangat optimal untuk diorkestrasi di *home lab* atau *server cluster* ringan, mengintegrasikan *container* langsung dengan antarmuka manajemen visual.
* **Database Management:** Menambahkan *Adminer* (port 4053) mempermudah inspeksi data di *production*.

### 3. Deployment Flow (CI/CD)

Proses pembaruan ke *server* berjalan sepenuhnya otomatis tanpa intervensi manual (Zero-Touch Deployment).

1. **Commit & Push:** Setiap perubahan yang di-*push* ke *branch* `main` memicu GitHub Actions.
2. **Build & Publish:** Runner GitHub akan melakukan *checkout* kode, melakukan proses kompilasi via Docker Buildx (menjalankan instalasi Composer & NPM), lalu mendorong (*push*) *image* final ke GHCR dengan *tag* `:latest`.
3. **Webhook Trigger:** Setelah *image* siap, GitHub Actions menembak *endpoint* Webhook n8n.
4. **Automated Rollout:** *Workflow* n8n di sisi *server* akan merespons webhook dengan menarik (*pull*) *image* terbaru dari GHCR dan me-*restart container* `app`.

### 4. Infrastructure Notes

Beberapa catatan teknis dan potensi perbaikan untuk infrastruktur Anda:

* **Permissions Handling (Smart CMD):** Perintah `CMD` pada Dockerfile sangat cerdas. Melakukan `chown` dan `chmod` saat *startup* memastikan masalah permission *storage* yang sering terjadi di Laravel teratasi otomatis, terlepas dari *host environment*-nya. Menjalankan *caching* (`config:cache`, `route:cache`) saat *startup* juga memastikan aplikasi langsung optimal begitu menyala.
* **Security Flag (Adminer):** *Service* Adminer terbuka di port `4053` secara publik di *production*. Ini memiliki risiko keamanan (brute-force). Disarankan untuk menutup port eksternal ini dan mengaksesnya via Reverse Proxy (seperti Nginx Proxy Manager/Cloudflare Tunnels) dengan lapisan autentikasi tambahan, atau hanya menyalakannya saat dibutuhkan.
* **Composer Flag:** Penggunaan `ENV COMPOSER_MEMORY_LIMIT=-1` dan `--no-dev --optimize-autoloader` sudah sangat tepat untuk mengamankan dan mengoptimalkan ukuran *image production*.

### Analisis Alur & Logika Eksekusi

1. **Pemicu (Webhook):**
Sistem mendengarkan *request* masuk dengan metode POST pada *path* spesifik (`78d8a4dc...`). Ini memastikan hanya entitas yang mengetahui URL unik ini (seperti GitHub Secrets Anda) yang bisa memicu *deployment*.
2. **Eksekusi Jarak Jauh (SSH + Tmux):**
Anda menggunakan perintah yang sangat strategis: `tmux new-session -d -s deploy_klikrental "cd /DATA/AppData/KlikRental/ && docker compose pull && docker compose down && docker compose up -d"`.
* **Penggunaan Tmux:** Menjalankan proses di *background* (*detached session*) adalah praktik yang sangat cerdas. Terkadang proses `docker compose pull` memakan waktu lama tergantung ukuran *image*. Jika dieksekusi langsung tanpa `tmux`, koneksi SSH dari n8n bisa mengalami *timeout* (terputus) sebelum proses selesai.


3. **Jeda Eksekusi (Wait Node):**
Sistem menunggu sekitar 1 menit sebelum melanjutkan proses. Ini adalah kompensasi logis karena eksekusi SSH diserahkan ke `tmux` (yang mana n8n akan menganggap proses SSH selesai seketika). Jeda ini memberi waktu bagi server untuk benar-benar mematikan dan menghidupkan kembali *container*.
4. **Multi-Channel Broadcasting:**
Setelah jeda selesai, sistem memparalelkan pengiriman notifikasi pembaruan ke Telegram (pesan ke Chat ID tertentu) dan WhatsApp (pesan ke Grup WA) lengkap dengan tautan produksi (`https://klikrental.widihhh.my.id/`).

### Catatan Infrastruktur & Praktik Terbaik

* **Optimalisasi Downtime Docker:**
Perintah `docker compose down && docker compose up -d` akan mematikan aplikasi sepenuhnya selama beberapa detik (atau menit) sampai *image* baru selesai ditarik dan dijalankan.
Untuk mencapai status yang lebih mendekati *Zero-Downtime*, Anda bisa merampingkan perintahnya menjadi:
`docker compose pull && docker compose up -d`
Docker Compose secara otomatis mendeteksi perubahan *image* dan hanya akan me-*recreate container* yang terpengaruh, sehingga waktu aplikasi mati akan jauh lebih singkat dibandingkan menggunakan `down` terlebih dahulu.
* **Validasi Keberhasilan (Error Handling):**
Saat ini, karena menggunakan `tmux`, n8n tidak mengetahui apakah proses `docker compose pull` atau `up` benar-benar berhasil atau mengalami *error* di tengah jalan. Notifikasi Telegram/WA akan selalu terkirim setelah 1 menit terlepas dari apakah *server* berhasil hidup atau gagal. Jika infrastruktur Anda berkembang, Anda bisa menambahkan *node* HTTP Request sebelum mengirim notifikasi untuk melakukan *ping* ke URL KlikRental (mengecek status HTTP 200) sebagai konfirmasi valid bahwa aplikasi benar-benar menyala.
